# LangChain 멀티 에이전트 (Multi-Agent Systems)

> 공식 문서: https://docs.langchain.com/oss/python/langchain/multi-agent

---

## 1. 개요

멀티 에이전트 시스템은 **복잡한 워크플로우를 전문화된 여러 에이전트가 협력**하여 처리하는 아키텍처다.

### 멀티 에이전트가 필요한 3가지 상황

| 동기 | 설명 |
|---|---|
| **컨텍스트 관리** | 단일 에이전트의 컨텍스트 윈도우를 초과하는 정보를 분산 처리 |
| **분산 개발** | 팀별로 독립적인 에이전트를 개발하고 조합 |
| **병렬화** | 서브작업을 동시에 실행해 속도 향상 |

> **주의:** 단순한 작업은 단일 에이전트가 더 효율적이다.
> 복잡성을 정당화할 충분한 이유가 있을 때만 멀티 에이전트를 사용하라.

---

## 2. 5가지 멀티 에이전트 패턴

### 2-1. Subagents (서브에이전트)

메인 에이전트가 서브에이전트들을 **도구처럼 호출**한다. 모든 라우팅이 코디네이터를 통해 이루어진다.

```
사용자
  ↓
[코디네이터 에이전트]
  ↓         ↓         ↓
[검색 에이전트] [계산 에이전트] [요약 에이전트]
```

**성능:** 단순 요청 4회 LLM 호출 / 멀티 도메인 5회 호출 + ~9K 토큰

**적합한 경우:** 분산 개발, 병렬화 필요 시

### 2-2. Handoffs (핸드오프)

에이전트들이 **서로 제어권을 넘기며** 협력한다. 직접 사용자와 상호작용 가능.

```
사용자
  ↕
[에이전트 A] ←→ [에이전트 B] ←→ [에이전트 C]
```

**성능:** 단순 요청 3회 / 반복 요청 2회 (상태 유지로 효율적)

**적합한 경우:** 다단계 추론, 직접 사용자 상호작용 필요 시

### 2-3. Skills (스킬)

하나의 컨트롤러 에이전트가 **전문 프롬프트/지식을 필요 시 로드**한다.

```
[컨트롤러 에이전트]
  ↓ 필요에 따라 로드
[법률 지식] [의료 지식] [금융 지식]
```

**성능:** 단순 요청 3회 / 멀티 도메인 3회 호출 (하지만 컨텍스트 누적으로 ~15K 토큰)

**적합한 경우:** 다양한 도메인 지식이 필요하나 분산 개발은 불필요한 경우

### 2-4. Router (라우터)

라우팅 컴포넌트가 **입력을 분류하고 적절한 에이전트에게 배분**한다.

```
사용자 입력
  ↓
[라우터]
  ↓         ↓         ↓
[에이전트 A] [에이전트 B] [에이전트 C]
  ↓         ↓         ↓
     [결과 합성]
```

**성능:** 단순 요청 3회 / 멀티 도메인 5회 + ~9K 토큰

**적합한 경우:** 병렬 실행, 명확한 입력 분류 가능한 경우

### 2-5. Custom Workflow (커스텀 워크플로우)

LangGraph로 구축한 **맞춤형 실행 흐름**. 결정론적 로직과 에이전트 동작을 혼합.

## 4. 성능 비교

### 단순 요청 (1회)

| 패턴 | LLM 호출 수 | 토큰 수 |
|---|---|---|
| Handoffs | 3 | 적음 |
| Skills | 3 | 적음 |
| Router | 3 | 적음 |
| Subagents | 4 | 적음 |

### 반복 요청 (상태 유지 효과)

| 패턴 | 첫 번째 | 두 번째 (반복) | 절감 |
|---|---|---|---|
| Handoffs | 3회 | 2회 | 33% ↓ |
| Skills | 3회 | 2회 | 33% ↓ |
| Subagents | 4회 | 4회 | 0% (상태 유지 없음) |

### 멀티 도메인 요청

| 패턴 | LLM 호출 | 토큰 수 | 특징 |
|---|---|---|---|
| Subagents | 5회 | ~9K | 병렬 처리, 독립 컨텍스트 |
| Router | 5회 | ~9K | 병렬 처리 |
| Handoffs | 7+회 | 14K+ | 순차 처리, 높은 비용 |
| Skills | 3회 | ~15K | 컨텍스트 누적 |

## 5. 핵심 설계 원칙

> **"멀티 에이전트 설계의 핵심은 컨텍스트 엔지니어링이다 — 각 에이전트가 무엇을 볼지 결정하는 것."**

패턴들은 **혼합 사용** 가능:
- 서브에이전트 아키텍처에서 라우터 패턴 활용
- 서브에이전트 자체가 스킬 패턴 사용
- 커스텀 워크플로우 노드 내에 핸드오프 패턴 포함

---

## 6. 언제 멀티 에이전트를 쓰지 말아야 하나?

- 단순한 QA → 단일 RAG 에이전트로 충분
- 도구 몇 개로 해결 가능한 경우 → 단일 에이전트
- 오버헤드(추가 LLM 호출, 지연시간) 감수할 가치가 없는 경우

> "적절한 도구와 프롬프트를 가진 단일 에이전트가 복잡한 작업을 처리하기에 충분한 경우가 많다."
